## 3) Архитектура (как требовалось в ТЗ)

### Что есть
- `user -> local state -> GPT(analysis/skill/crisis) -> template response -> SQLite events -> background sheets sync` реализовано в SKILLER-ветке.

### Что мешает полному соответствию
- Параллельно остаётся legacy-оркестрация (большой `KNOWN_STAGES` + старые сценарии в `bot.py`).
- Это не всегда нарушает ТЗ, но снижает «жёсткость» новой архитектуры.

---

## 4) База данных против ТЗ

### Соответствует
- Таблицы `users`, `user_state`, `patterns`, `events` существуют.
- Поля SKILLER в `users` (day/mode/skill/style/free_mode/checkin flags) есть.
- `events` поддерживает `event_name`, `event_data`, `created_at`, `synced`.

### Нюанс
- `events` содержит legacy-колонки (`ts`, `stage`, `event`, `meta`) — это допустимо, но схема стала гибридной.

---

## 5) GPT-правила из ТЗ

### Соответствует
- Есть отдельные prompt-функции: `get_analyze_state_prompt`, `get_select_skill_prompt`, `get_crisis_check_prompt`.
- В prompt явно зафиксированы ограничения: только JSON, без диагнозов, без «свободного ответа пользователю», лимит по short/long analysis.

### Нюанс
- В runtime остаются локальные текстовые fallback-ветки и legacy тексты; они не GPT, но это не полностью «чистая» template-first модель.

---

## 6) UX-цепочка и Action loop

### Соответствует
- Есть путь: `/start -> проблема -> short analysis -> Подробнее/Давай действие -> навык -> Сделал/Не сделал -> downscale -> вечер -> day3 offer -> payment stub -> free mode`.
- Кнопки и ключевые тексты соответствуют ТЗ-паттерну.

### Нюанс
- Часть текста ещё хардкодится в `skiller_runtime.py`/`bot.py`, а не только в `templates.py`.

---

## 7) Free/Paid, платежная заглушка и /stats

### Соответствует
- Платежная заглушка `€20/€40/Подумаю` есть.
- Логи `payment_click_20`, `payment_click_40`, `payment_declined_soft`, `payment_completed` есть.
- `/stats` реализован, с ACL через `ADMIN_IDS` и нужными метриками.

### Нюанс
- Управление free_mode частично разнесено между runtime и payment-хендлером (лучше централизовать).

## 11) Ответ на вопрос «всё работает?»

Коротко: **не всё**.

- Основной продуктовый сценарий в новой SKILLER-ветке выглядит рабочим по структуре (state machine + action loop + offer/free mode).
- Но автотесты сейчас падают с `ModuleNotFoundError: bot`, поэтому формально утверждать «всё работает» нельзя до починки тестового импорта/окружения.
- Также остаётся архитектурный риск из-за параллельной legacy-ветки в `bot.py`.
